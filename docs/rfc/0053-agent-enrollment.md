# RFC-0053: Agent Enrollment

**Status**: Active — v2.0.0 Phase A  
**Depends on**: [RFC-0050](0050-v2-architecture-rebaseline.md), [RFC-0052](0052-backend-core-api.md)  
**Feeds**: RFC-0056

---

## Purpose

Define how an Agent is registered with the Backend, how trust is established, and what the Agent lifecycle looks like from the Backend's perspective.

Agents are Backend-managed infrastructure. The Launcher has no visibility into enrollment.

---

## Enrollment flow

```text
Backend operator generates one-time token
           ↓
Operator installs Agent on server host
(e.g. curl https://.../install.sh | bash)
           ↓
Agent calls POST /api/v1/agents/register
with enrollment token + serverId
           ↓
Backend validates token, marks used
           ↓
Backend returns agent access token
           ↓
Agent stores token securely
           ↓
Agent begins heartbeat + manifest cycle
```

The one-time enrollment token is:
- Generated by a Backend operator or admin API
- Single-use
- Short-lived (default 24h expiry)
- Scoped to a specific `serverId`

---

## Agent lifecycle states

```text
Pending
  ↓ register() called with valid token
Registered
  ↓ first heartbeat received
Healthy
  ↓ heartbeat interval missed (configurable, default 2× interval)
Offline
  ↓ operator revokes
Revoked (terminal)
```

State transitions:

| From | To | Trigger |
|------|----|---------|
| Pending | Registered | Successful `POST /agents/register` |
| Registered | Healthy | First `POST /agents/heartbeat` |
| Healthy | Offline | Missed heartbeat deadline |
| Offline | Healthy | Heartbeat resumes |
| Any | Revoked | Operator action |

Revoked agents cannot re-register with the same token. A new enrollment token must be issued.

---

## Heartbeat contract

Agents MUST send a heartbeat at a regular interval (default: every 30 seconds).

Backend marks an Agent Offline if no heartbeat is received within `2 × interval` (default: 60 seconds).

Heartbeat payload: see RFC-0052 `POST /api/v1/agents/heartbeat`.

The Backend aggregates heartbeat data to:
- Update server `status` and `playerCount` in the registry
- Determine content availability for `POST /join` resolution

---

## Manifest submission

After each mod change detected on the server filesystem, the Agent:

1. Rebuilds the manifest (`ModEntry[]` from discovered mods)
2. Submits `PUT /api/v1/agents/manifest`
3. Backend versions and stores the manifest
4. New manifest becomes active for subsequent `POST /join` calls

Manifest submission is independent of heartbeat. Both may be in-flight simultaneously.

---

## Content availability

When the Backend needs to resolve content for a `POST /join` request:

1. Backend checks its own content-addressable store (by SHA256)
2. If missing, Backend requests the blob from the relevant Agent
3. If Agent is Offline or content unavailable, Backend falls back to SteamCMD (last resort)

The Agent exposes content on a Backend-internal endpoint (not in the public API).

---

## Security

- Enrollment tokens are single-use and short-lived
- Agent access tokens have a long TTL (e.g. 1 year) but can be revoked
- Agent tokens are scoped: an Agent for `serverId=demo-survival` cannot submit manifests for other servers
- All Agent-to-Backend communication is over HTTPS

---

## Non-goals

- Agent auto-update mechanism (v2.1+)
- Multi-server per Agent (one Agent per server host in v2.0.0)
- Agent UI or management dashboard
- Agent-to-Agent communication
